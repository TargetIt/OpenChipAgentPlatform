# AI Agent 工程师交流问题清单

场景：晚上和一位档案管理公司的 Agent 工程师交流。  
目标：这不是技术面试，而是通过请教、交流和产品讨论，判断他是否真的理解企业级 Agent 工程，是否适合参与 OpenChipAgentPlatform / 芯片验证 Agent 方向。

## 使用方式

不要按面试题逐条拷问。建议按下面节奏交流：

1. 先请他介绍他们现在的档案管理 Agent 产品。
2. 顺着他的产品追问工程实现、可靠性、审计和客户落地。
3. 再请他反过来评价我们的芯片验证 Agent 想法。
4. 最后观察他是否能把经验抽象成 Harness、workflow、failure taxonomy、human review、证据链。

重点不是看他是否懂 EDA，而是看他是否懂“可靠 Agent 产品怎么落地”。

## 一、先看他怎么理解 Agent

1. 你觉得 Agent 和普通 Chatbot / Copilot 最大的区别是什么？
2. 你做过的 Agent 里，最长链路的任务是什么？从目标到完成经历了哪些步骤？
3. 一个 Agent 什么时候应该自己继续执行，什么时候应该停下来问人？
4. 你怎么看“模型能力”和“Agent 工程能力”的关系？
5. 如果模型很强，但 Agent 经常跑偏，你会从哪些地方排查？

## 二、围绕他们的档案管理产品请教

1. 你们档案管理 Agent 主要解决哪类用户问题？是检索、归档、分类、审核，还是流程自动化？
2. 你们的 Agent 是偏“问答助手”，还是会主动执行任务？
3. 用户给一个模糊需求时，你们 Agent 怎么判断要追问，还是直接执行？
4. 档案管理里应该有很多格式和字段不统一的问题，你们怎么做结构化抽取？
5. 你们如何处理 OCR 错误、文档缺页、字段冲突、重复档案这些脏数据？
6. 你们系统里 Agent 的“完成标准”是什么？用户怎么知道它真的完成了？
7. 你们有没有遇到 Agent 看似完成、实际漏处理文件的情况？怎么防？
8. 你们有没有做任务状态管理？比如一个归档任务跑到一半失败了，怎么恢复？
9. 你们有没有做审计日志？能不能回看 Agent 每一步为什么这么做？
10. 你们现在最难落地的客户场景是什么？

## 三、看他有没有真实工程经验

1. 你做过哪些工具调用？比如 shell、浏览器、数据库、代码仓库、API、CI、文档系统？
2. 你怎么设计工具调用的输入输出，避免模型乱填参数？
3. Agent 执行过程中失败了，你怎么保存状态，让它能恢复或让另一个 Agent 接手？
4. 你有没有做过长任务执行？比如跑几十分钟、多轮修改、多次验证。
5. 你会怎么记录每一步的证据？比如命令、日志、文件、diff、失败原因。
6. 你们有没有做多 Agent？如果做了，任务怎么拆、结果怎么合并？
7. 你们有没有把某类失败沉淀成固定 workflow 或 skill？
8. 有没有某个功能你们一开始以为模型能搞定，后来发现必须工程化？

## 四、围绕可靠性和企业落地追问

1. 档案管理应该对准确性要求很高，你们怎么降低幻觉？
2. Agent 给出答案时，是不是必须引用原文证据？
3. 如果多个文档证据冲突，Agent 怎么处理？
4. 你们有没有 human review 机制？哪些操作必须人确认？
5. Agent 有没有权限边界？比如哪些文件能看、哪些不能看、哪些动作不能自动做？
6. 如果 Agent 误删、误归档、误分类，你们怎么追责和回滚？
7. 你们怎么做测试集？是人工标注，还是从真实客户数据抽样？
8. 你们怎么衡量产品效果？准确率、节省时间、处理量、客户满意度分别怎么看？
9. 你们上线后有没有持续学习机制？失败案例怎么沉淀？
10. 你们有没有做过“失败类型分类”？比如 OCR 错误、权限错误、字段缺失、用户意图不清。

## 五、让他给我们的芯片验证 Agent 提建议

1. 如果我们做“芯片验证 Agent”，你第一反应觉得最大风险是什么？
2. 你们在档案管理里怎么做“证据链”？这个经验能不能迁移到 EDA log、波形、覆盖率？
3. 我们客户是验证工程师，非常挑剔。你觉得企业客户接受 Agent 的前提是什么？
4. 如果 Agent 要处理很长的仿真 log，你会借鉴你们处理长文档的方法吗？
5. 你会怎么设计一个“可审计的 Agent run 记录”？
6. 我们要防止 Agent 为了通过验证而弱化测试。你们有没有类似“不能为了完成任务而改规则”的机制？
7. 如果我们第一版只做 Harness，不训练模型，你觉得合理吗？为什么？
8. 你觉得我们早期应该先做哪种 demo，才能证明不是玩具？
9. 如果你负责我们这个产品的 Agent 架构，前三周你会先搭什么？
10. 你觉得档案管理 Agent 和芯片验证 Agent 最大的共性、最大的不共性是什么？

## 六、专门考察 Harness 理解

1. 你怎么理解 Harness？它和 Agent runtime 是什么关系？
2. 如果我要做一个“AI Agent + EDA 工具”的系统，你会先做模型、runtime，还是 Harness？为什么？
3. 一个好的 Agent Harness 应该包含哪些东西？
4. 你会怎么设计 failure taxonomy？
5. 你会怎么把一次失败修复沉淀成 reusable skill？
6. 如果 Agent 处理一批文件，中间失败，你们怎么恢复现场？这个机制能不能迁移到芯片验证 run？
7. 如果另一个 Agent 或工程师要接手当前任务，你觉得 run 记录里必须保留哪些东西？

## 七、看产品感觉和合作方式

1. 如果客户是非常挑剔的专业工程师，你觉得产品第一版必须做到什么程度？
2. 你怎么判断一个 Agent 产品是 demo，还是真能落地？
3. 你喜欢从 0 到 1 搭系统，还是在已有系统上优化？
4. 你能接受和领域专家一起反复抠细节吗？举个例子。
5. 如果你加入这个方向，前三个月你觉得最应该做出什么结果？
6. 用户不信任 Agent 时，你们怎么设计交互让他重新信任？
7. 你觉得 Agent 产品最容易被高估的地方是什么？
8. 你觉得 Agent 产品最容易被低估的工程工作是什么？

## 八、建议重点追问的几个问题

这些问题最容易看出他是不是只会 prompt，还是做过真正企业级 Agent 产品：

1. 你做过的最长 Agent 链路是什么？
2. 失败之后怎么恢复？
3. 怎么防止 Agent 为了完成任务而作弊？
4. 你怎么定义 done？
5. 如何把失败沉淀成 skill？
6. 你们怎么让客户相信 Agent 没有漏处理？
7. 如果你来做芯片验证 Agent，你第一版会先做哪个闭环？
8. 你会如何接入一个完全陌生的 EDA CLI？

## 九、观察要点

如果他的回答中自然出现这些关键词，说明他可能有真实经验：

- 证据链
- 状态机
- 任务恢复
- 权限边界
- 人工审核
- 失败分类
- 可观测性
- 长上下文切分
- 引用来源
- 数据质量
- 回滚
- 评测集
- 端到端流程
- 不让 Agent 自己改规则
- 从失败中沉淀 workflow / skill

## 十、判断信号

### 正向信号

- 能讲清楚一个真实长链路 Agent 任务。
- 反复强调证据、状态、恢复、审计和权限。
- 能区分模型问题、工具问题、数据问题、流程问题。
- 能承认 Agent 需要 human review，而不是一味追求全自动。
- 能把档案管理经验抽象迁移到芯片验证场景。
- 关注客户信任、企业落地和产品可维护性。

### 风险信号

- 只讲 prompt，不讲状态管理和工具执行。
- 把 Agent 等同于聊天机器人或 RAG 问答。
- 过度承诺“模型会自己解决”。
- 不重视权限、审计、回滚和失败恢复。
- 对“不能为了通过任务而改规则”没有意识。
- 无法讲出真实失败案例。
- 没有 done condition 的概念。

## 十一、推荐开场话术

可以这样开始：

```text
我们不是做技术面试，主要想请教你们做档案管理 Agent 的真实经验。
我们现在在思考一个芯片验证 Agent，和档案管理有一些共性：都需要处理长文档/长日志，都要证据链，都对准确性和企业落地要求很高。
所以我想从你们产品怎么做、怎么避免出错、怎么让客户信任这几个角度聊一聊。
```

## 十二、推荐收尾问题

最后可以问：

```text
如果你站在我们的角度，觉得我们这个 AI Agent + 芯片验证方向，第一步最应该证明什么？
你觉得什么样的 demo 才能让专业验证工程师相信这不是玩具？
如果你加入，你最想先解决哪个工程问题？
```
