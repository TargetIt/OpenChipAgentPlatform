# OpenChipAgentPlatform 投资人/合伙人交流材料草稿

版本：2026-06-27

定位：这不是最终路演稿，而是第一版沟通底稿。目标是把行业背景、客户痛点、产品切入点、发展路径和启动团队画像讲清楚。

## 0. 一句话

OpenChipAgentPlatform 要做的是面向芯片设计验证场景的 AI Agent + EDA Harness 平台：让 AI 不只是“写代码”，而是在真实 EDA 工具、验证流程和质量出口标准约束下，持续生成、运行、调试、收敛验证任务。

核心判断：

- 大模型会显著改变芯片设计流程，但芯片行业不能接受“看起来对”的幻觉产物。
- 真正有价值的 AI Agent 必须被 EDA 工具链、验证证据和工程规范约束。
- 验证是最适合作为第一落点的环节，因为它既是质量核心，也是效率瓶颈。

## 1. 传统芯片产业链：从规格到硅片

### Slide 1：传统芯片怎么被设计出来

传统数字芯片设计大致经历：

```text
市场/产品需求
  -> 芯片规格 Spec
  -> 架构设计 Architecture
  -> RTL 设计
  -> 功能验证
  -> 逻辑综合
  -> DFT / 可测性设计
  -> 形式验证 / 等价性检查
  -> 后端物理设计
  -> 时序/功耗/面积收敛
  -> Signoff
  -> Tape-out
  -> 晶圆制造
  -> 封装测试
  -> 系统验证 / 量产
```

这个流程的本质是：先把“想要什么芯片”变成精确规格，再把规格变成 RTL，再证明 RTL 行为正确，然后把 RTL 变成可制造的物理版图，最后交给晶圆厂制造。

### Slide 2：传统环节中的三类核心角色

```text
芯片设计公司
  负责定义产品、架构、RTL、验证、后端集成和最终 tape-out 责任

EDA 公司
  提供设计、验证、综合、布局布线、仿真、签核等工具

制造/封测公司
  负责晶圆制造、工艺实现、封装、测试和量产能力
```

传统时代，芯片设计公司是流程的组织者和主要知识载体。EDA 工具非常关键，但更多是工程师手里的工具。制造公司也极其重要，但它主要承接已经完成签核的设计。

### Slide 3：传统价值分布的朴素理解

传统模式下，价值和责任大致集中在：

- 设计公司：产品定义、架构判断、工程实现、项目管理、最终质量责任。
- EDA 公司：提供工具，但很多智能仍然在工程师脑中。
- 制造公司：提供工艺、良率和产能。

这意味着传统流程高度依赖大量高水平设计/验证工程师。工具重要，但工具本身不直接承担“工程师式的连续任务”。

## 2. AI Agent 时代：芯片设计链路会重新分工

### Slide 4：AI Agent 不是另一个编辑器插件

代码补全、问答助手、普通 Copilot 对芯片设计有帮助，但不足以改变产业链。

真正改变流程的是 Agent：

```text
给定目标
  -> 理解设计上下文
  -> 制定验证/实现计划
  -> 调用 EDA 工具
  -> 读取日志/波形/覆盖率
  -> 定位失败
  -> 修改代码或测试
  -> 反复运行直到达到出口标准
```

也就是说，Agent 的价值不在于“写一段 RTL”，而在于“围绕工程目标持续闭环”。

### Slide 5：新的产业链假设

在 AI Agent 时代，EDA 公司和 EDA-like 平台的重要性会上升。

原因是：

- Agent 需要工具反馈作为确定性约束。
- Agent 需要把日志、波形、覆盖率、仿真结果、综合结果都纳入决策。
- Agent 需要行业知识、流程规范和质量出口，而这些天然靠近 EDA 工具链。
- 设计公司仍然定义产品和做最终签核，但大量执行型工作会被 Agent + EDA 自动化吸收。

新的价值分布可能变成：

```text
AI-native EDA / Agent 平台
  承担越来越多工程执行、质量收敛、知识沉淀

制造公司
  继续承担工艺、制造、封测、良率和产能壁垒

设计公司
  更聚焦产品定义、架构选择、差异化 IP、最终 signoff
```

这不是说设计公司不重要，而是设计公司的组织形态会改变：从“人海工程执行”转向“少数高水平工程师 + Agent 工程系统”。

### Slide 6：为什么这件事现在发生

几个条件同时成熟：

- 大模型已经具备读写代码、读文档、规划任务、调用工具的基础能力。
- 开源 Agent 框架和通用代码 Agent 正在快速成熟。
- EDA 工具普遍已有 CLI、日志、报告、脚本化能力，天然适合 Agent 调用。
- 芯片复杂度继续上升，验证和调试压力越来越大。
- 市场窗口要求更快 tape-out，时间就是竞争力。

## 3. 客户痛点：设计公司真正需要什么

### Slide 7：客户不是普通开发者，是极挑剔的硬件工程师

我们的早期客户是：

- 数字 IC 设计工程师
- 验证工程师
- 架构/前端负责人
- EDA/验证平台团队

这些客户本身就是工程专家。他们不会被“AI 看起来很聪明”打动。他们关心的是：

- 能不能跑在真实项目里？
- 能不能减少真实工作量？
- 出错时能不能解释清楚？
- 能不能保留证据？
- 会不会偷偷弱化验证标准？
- 能不能接入公司已有工具链？

这里容不得半点沙子。产品必须经得起工程师反复挑刺。

### Slide 8：第一痛点是质量

芯片不同于普通软件。软件 bug 可以上线后热修复，芯片 bug 一旦进入硅片，代价可能是：

- 重新流片；
- 项目延期数月；
- 客户窗口丢失；
- 大量验证、后端、软件和系统团队返工；
- 品牌和客户信任受损。

所以设计公司不会因为 AI 快就接受低质量。芯片 AI Agent 的第一原则是：

```text
没有证据的“完成”不算完成。
没有工具验证的代码不算可靠。
不能复现的结果不算工程结果。
```

### Slide 9：第二痛点是速度

芯片开发周期长，且市场窗口非常敏感。

AI 芯片、车规芯片、通信芯片、边缘计算芯片都面临类似压力：晚几个月上市，可能意味着错过客户导入窗口。

如果 Agent 能把部分验证计划、测试生成、回归调试、覆盖率分析、日志定位从“天/周”压缩到“小时/分钟”，它带来的价值不是节省几个人力，而是提前上市、减少项目风险。

### Slide 10：为什么通用 Agent + EDA CLI 还不够

今天已经可以用 OpenCode、Claude Code、Codex 加 EDA CLI 做一些自动化。

但直接组合会遇到问题：

- 没有统一目标格式，Agent 容易跑偏。
- 没有强制证据链，容易“写完就停”。
- 没有验证出口标准，容易把测试跑过当成质量闭环。
- 没有失败分类和修复策略，调试不可复用。
- 没有波形、覆盖率、日志、断言、UVM 等硬件验证语义。
- 没有公司级可审计记录，难以进入严肃设计流程。

所以我们要做的不是再造一个聊天机器人，而是做芯片验证 Agent 的工程 Harness。

## 4. 为什么从验证切入

### Slide 11：验证是芯片质量的核心

验证回答的问题是：这个芯片是否真的按照规格工作？

验证覆盖：

- 功能测试；
- UVM testbench；
- directed/random tests；
- assertion；
- reference model；
- coverage closure；
- regression；
- debug；
- waveform/log 分析。

如果 RTL 是“生产逻辑”，验证就是“质量防线”。AI 可以生成 RTL，但如果没有验证约束，生成越快，风险也可能越快扩散。

### Slide 12：验证也是最典型的效率瓶颈

公开行业资料显示，功能验证长期占据芯片工程周期的大头。Siemens / Wilson Research Group 2024 年报告提到，IC/ASIC 首次硅成功率下降到 20 年低点，只有 14% 项目达到 first-silicon success。ChipAgents 等公开材料也把验证描述为 60%-70% 工程时间消耗的瓶颈。

这说明验证不是边角环节，而是影响成本、周期和质量的主战场。

### Slide 13：AI 最大矛盾：生产力强，但不确定

AI 的优势：

- 能快速读规格；
- 能生成测试代码；
- 能生成 assertion；
- 能总结日志；
- 能提出 debug 假设；
- 能快速迭代。

AI 的问题：

- 可能幻觉；
- 可能漏掉边界条件；
- 可能生成看似合理但语义错误的验证代码；
- 可能为了通过测试而弱化测试；
- 可能无法意识到覆盖率不足。

我们的核心产品问题就是：

```text
如何释放 AI 的生产力，同时用 EDA 工具、验证方法学和证据链把质量确定性拉上来？
```

### Slide 14：公开对标

ChipAgents / Alpha Design AI：

- 方向：agentic AI chip design environment。
- 公开定位：设计和验证自动化，覆盖规格、RTL、testbench、debug、verification。
- 公开目标：提升 RTL 设计和验证效率，宣称最高 10x productivity。

智维创芯 / ChatDV：

- 方向：面向数字芯片验证的大模型智能体。
- 公开描述：覆盖测试生成、断言生成、参考模型构建、自动调试。
- 公开案例：媒体报道提到其在 RISC-V CPU 设计 Rocket/BOOM 中发现此前未发现的功能 bug，并宣称覆盖率和效率提升。

Cadence / Synopsys：

- 方向：将 AI Agent 嵌入既有 EDA 工具体系。
- 关键启发：最终一定要绑定真实 EDA 引擎、仿真、覆盖率、签核和工程证据。

我们的判断：这个方向不是“要不要做”，而是“谁能真正落地到工程流程”。

## 5. 我们要做的产品

### Slide 15：产品定位

OpenChipAgentPlatform 的第一阶段产品定位：

```text
AI Verification Agent + Harness Engineer
```

它不是单纯模型，也不是单纯 EDA 工具，而是把通用 Agent、大模型、EDA CLI、验证流程、日志/波形/覆盖率分析、质量出口标准整合起来。

一句话：

```text
让 AI 像一个初级到中级验证工程师一样工作，并让资深工程师可以审计、接管、指导它。
```

### Slide 16：产品核心闭环

```text
输入：规格 / RTL / 历史测试 / Bug 描述
  -> 解析规格和设计意图
  -> 生成验证计划
  -> 生成 testbench / assertion / reference model
  -> 调用仿真器和 EDA 工具
  -> 读取 log / waveform / coverage
  -> 分类失败
  -> 智能 debug
  -> 修改测试或建议 RTL 修复
  -> 回归运行
  -> 达到出口标准后生成验证报告
```

这个闭环必须每一步都有证据，不能只靠模型自信。

### Slide 17：产品模块

1. Spec Understanding Agent  
   读取自然语言规格、接口说明、已有 RTL，抽取端口、协议、时序、reset、corner cases。

2. Verification Plan Agent  
   生成验证计划，包括 directed tests、random tests、coverage points、assertions、负例测试。

3. Testbench Generation Agent  
   生成 SystemVerilog/UVM 或 cocotb testbench，构建 driver、monitor、scoreboard、reference model。

4. Execution Agent  
   调用仿真器、lint、coverage、formal、脚本化 EDA 工具，保存日志和产物。

5. Debug Agent  
   读取编译错误、仿真失败、assertion fail、波形、coverage hole，定位失败原因。

6. Harness Engineer  
   维护目标、流程、schema、质量出口、证据链、review gate 和可复现 run。

7. Human-in-the-loop Review  
   在修改规格、弱化测试、接受 warning、改变协议语义等关键节点要求工程师确认。

### Slide 18：验证出口标准

我们的产品必须明确“什么时候算完成”：

- 编译通过不等于完成；
- 仿真通过不等于完成；
- 覆盖率高但关键场景漏掉也不等于完成；
- AI 说“应该没问题”更不等于完成。

一个验证任务完成至少要满足：

- 规格被结构化；
- 验证计划覆盖关键功能和 corner cases；
- testbench/reference model/assertion 可运行；
- regression 通过；
- coverage 达到目标或明确说明缺口；
- 失败和修复有记录；
- 关键变更经过 human review；
- 最终报告可复现。

### Slide 19：我们和通用代码 Agent 的差异

| 维度 | 通用代码 Agent | OpenChipAgentPlatform |
| --- | --- | --- |
| 目标 | 写代码/改代码 | 达成验证目标 |
| 反馈 | 编译/测试为主 | 仿真、覆盖率、日志、波形、EDA 报告 |
| 成功标准 | 测试通过 | 质量出口 + 证据链 |
| 调试 | 读错误日志 | 读 EDA log、波形、assertion、coverage |
| 安全 | 通用代码审查 | 不弱化验证、不改规格、不跳过 signoff |
| 记忆 | 代码上下文 | 验证知识、失败分类、修复策略、项目 run 记录 |

## 6. 发展路径

### Slide 20：阶段一，开源 Agent + Harness

目标：最快验证产品假设。

做法：

- 利用 OpenCode / Claude Code / Codex 等通用 Agent；
- 建立芯片验证 Harness；
- 先支持开源工具链：Verilator、cocotb、Yosys、开源 RTL；
- 跑通小模块验证任务；
- 沉淀 schema、workflow、failure taxonomy、skill memory。

关键指标：

- 能否稳定完成模块级验证任务；
- 能否自动分类失败并迭代；
- 工程师是否愿意继续使用；
- 是否能真实减少重复劳动。

### Slide 21：阶段二，Harness + 闭源 EDA 工具

目标：证明它能进入真实工业流程。

做法：

- 接入商业仿真器、lint、coverage、formal、UVM 环境；
- 与已有 EDA CLI 和 regression 系统协同；
- 在真实设计公司的小范围项目中试点；
- 重点验证质量、可审计性、安全边界。

关键指标：

- 在真实项目中减少 debug/testbench/coverage closure 时间；
- 不破坏既有流程；
- 能保留完整证据链；
- 工程师愿意把部分任务交给 Agent。

### Slide 22：阶段三，自有 Agent + Harness，模型使用第三方 API

目标：形成完整产品。

做法：

- 自研 Agent runtime；
- 自研 chip verification workflow engine；
- 自研上下文管理、工具调用、权限、审计和任务队列；
- 底层模型先使用第三方 API；
- 产品形态面向团队，而不是单个工程师插件。

关键指标：

- 交付企业试用版；
- 支持项目级 run 管理；
- 支持多 Agent 协作；
- 支持公司级权限、审计、知识库和私有数据隔离。

### Slide 23：阶段四，自有模型/私有部署 + 整体产品

目标：企业级规模化交付。

做法：

- 支持私有化部署；
- 训练或微调面向芯片验证的专用模型；
- 使用客户私有知识库和历史工程数据做 RAG/记忆增强；
- 提供从任务输入到验证报告的开箱即用平台。

关键指标：

- 数据安全满足大客户要求；
- 模型对 EDA/验证语义理解显著优于通用模型；
- 客户可以将其纳入标准验证流程。

## 7. 启动阶段人才画像

### Slide 24：第一优先级是 Agent 人才

我们现在最需要的是 Agent 工程人才，而不是纯模型研究人才。

原因：

- 早期胜负手不是训练大模型，而是把现有模型放进真实工程闭环。
- 芯片验证场景的难点在工具调用、上下文、错误恢复、证据链、workflow 和 human review。
- 只会调 prompt 不够，必须能把 Agent 变成可靠工程系统。

理想画像：

- 熟悉 LLM Agent 架构；
- 能做工具调用、长任务执行、状态管理；
- 能设计 schema/workflow/harness；
- 能处理日志、失败分类、自动重试；
- 有代码工程能力，能把 demo 做成产品；
- 对芯片/EDA 不一定一开始很深，但必须愿意深入硬件工程流程。

### Slide 25：第二优先级是芯片验证专家

Agent 人才必须和验证专家紧密迭代。

验证专家负责定义：

- 什么是正确的验证任务；
- 什么是不能自动改的边界；
- 什么样的 coverage 才有意义；
- 哪些 debug 结论可信；
- 哪些操作必须人工 review；
- 企业真实流程怎么接入。

没有验证专家，Agent 很容易做成“看起来能跑”的玩具。

### Slide 26：第三阶段再强化大模型人才

大模型人才也重要，但可以晚一点重投入。

早期更重要的是：

- 选好模型；
- 建好 Agent Harness；
- 积累高质量 run 数据；
- 形成 failure/repair 数据集；
- 跑出真实客户价值。

等到数据和产品闭环成立，再做：

- 专用模型微调；
- EDA log/RTL/verification 语料训练；
- 私有部署推理优化；
- 安全和成本优化。

## 8. 结尾页：为什么我们有机会

OpenChipAgentPlatform 的机会来自三个交叉点：

```text
芯片验证是刚需
  + AI Agent 能执行长链路工程任务
  + EDA 工具天然提供确定性反馈
```

我们的策略不是一开始就造完整 EDA，也不是一开始就训练大模型，而是先做最现实、最能落地的中间层：

```text
Harness Engineer + AI Verification Agent
```

它的价值是把 AI 的不确定生产力放进芯片验证的确定性工程流程里。

如果这个闭环成立，它可以从模块级验证扩展到 IP 级验证、SoC 子系统验证、覆盖率收敛、debug 自动化，最终成为 AI-native EDA 平台的一部分。
